# 보안기능 구현명세서 — 3장 기능 설명 (living)

이 문서는 **손으로 쓰는 서술(4단 골격)** + **자동 상태표**로 구성된다.
- 서술 부분은 언제든 직접 수정 가능(코드 작업 중에도).
- 맨 아래 `AUTO-STATUS` 블록은 `node sfr/sync.mjs --write` 가 코드의 `SFR:<id>` 태그를 스캔해 갱신한다(직접 수정 금지).
- 완성되면 rhwp-bridge 로 `.hwpx` 변환 → ① 구현명세서 3장에 삽입.

> 각 보안기능은 가이드 4단 골격으로: **해당 요구사항 / 설명 / 동작 순서도 / 동작 절차**.

---

## 3.2 보안기능 개요

제품(RPi5+IMX219, 순수 스트리밍)의 보안기능을 최소 단위로 식별. 접속평면·인증·영상보안·자체보호·감사·업데이트·암호로 구분하며, 각 기능은 아래 3.3에 개별 기술.

## 3.3 관리서버(카메라 본체)의 보안기능

### 3.3.1 관리 접속경로 및 방법 (SFR:3.3.1 / 4.1.1)

| 구분 | 내용 |
|------|------|
| 해당되는 국가용 보안요구사항 | 3.3.1(관리 IP 제한), 4.1.1(관리접속 전송보호) |
| 보안기능 설명 | 관리 접속은 HTTPS(TLS1.2+)·SSH만 허용하며, nftables default-deny + 관리 IP allowlist로 제한한다. 평문 HTTP·telnet 포트는 이미지에 없다. |
| 보안기능 동작 순서도 | 〈그림〉 |
| 보안기능의 동작 | (절차 1) 접속 요청 소스 IP를 nftables `mgmt_allow` set과 대조. (절차 2) 미등록이면 drop. (절차 3) 등록 IP만 443/22 허용 후 TLS/SSH 핸드셰이크. |

*(구현 위치: `board/rpi5-secure/rootfs-overlay/etc/nftables.conf`, `.../etc/ssh/sshd_config`)*

### 3.3.2 기본 계정·패스워드 강제 변경 (SFR:3.4.1 / 3.4.2)

| 구분 | 내용 |
|------|------|
| 해당되는 국가용 보안요구사항 | 3.4.1(기본계정 강제변경·사용중지), 3.4.2(default PW 강제변경) |
| 보안기능 설명 | 공장초기화 상태(`/etc/.factory-state`)로 출하되며, 최초 접속 시 기본 패스워드를 강제 변경하기 전에는 보안관리 기능이 잠긴다. |
| 보안기능 동작 순서도 | 〈그림〉 |
| 보안기능의 동작 | (절차 1) 부팅 시 factory-state 플래그 확인. (절차 2) 존재 시 PW 변경 화면만 제공, 그 외 기능 차단. (절차 3) 변경 완료 후 플래그 제거·정상 기능 개방. |

*(구현 위치: `package/cctv-secd/src/main.c` `provision_firstboot()`)*

### 3.3.3 자체시험 및 무결성 검증 (SFR:5.1.1 / 5.2.1 / 5.2.3)

| 구분 | 내용 |
|------|------|
| 해당되는 국가용 보안요구사항 | 5.1.1(자체시험), 5.2.1(무결성 검증), 5.2.3(결과 확인) |
| 보안기능 설명 | 부팅 시와 주기적(그리고 관리자 요청 시)으로 dm-verity 기반 rootfs 무결성과 설정파일 HMAC을 검증한다. 결과는 웹UI·감사로그로 확인한다. |
| 보안기능 동작 순서도 | 〈그림〉 |
| 보안기능의 동작 | (절차 1) 부팅 시 dm-verity가 rootfs 해시트리 검증. (절차 2) cctv-secd가 설정파일 HMAC 검증(crypto_hal_hmac). (절차 3) 실패 시 안전모드·경고·감사기록(SFR:5.2.4). |

*(구현 위치: `package/cctv-secd/src/main.c` `self_test()`, `board/.../genimage.cfg` verity)*

### 3.3.4 영상 프로토콜 인증 및 전송 보호 (SFR:1.1.1 / 1.2.1)

| 구분 | 내용 |
|------|------|
| 해당되는 국가용 보안요구사항 | 1.1.1(표준 프로토콜 사용자 인증), 1.2.1(영상 전송 암호통신) |
| 보안기능 설명 | IMX219 영상은 RTSP 서버(mediamtx)를 통해 스트리밍되며, 익명 접근을 차단하는 사용자 인증(internal auth)과 RTSPS(RTSP over TLS, `rtspEncryption: strict`)를 강제한다. 평문 RTSP·UDP 전송은 비활성화된다. RPi5는 HW 인코더가 없어 libcamera→gstreamer 소프트웨어 H.264로 인코딩한다. |
| 보안기능 동작 순서도 | 〈그림〉 |
| 보안기능의 동작 | (절차 1) 클라이언트가 RTSPS(TLS)로 접속, 평문 RTSP는 거부. (절차 2) internal auth로 사용자(viewer/publisher) 인증, 실패 시 스트림 미제공. (절차 3) 인증된 세션에만 TLS 암호화된 영상 전송. (절차 4) 서버 인증서 부재 시 기동 보류(평문 폴백 금지). |

*(구현 위치: `package/rtsp-server/mediamtx.yml`, `.../S65rtsp-server`, `board/.../nftables.conf`)*

### 3.3.5 업데이트 보호 (SFR:6.1.1 / 6.1.2 / 6.1.3)

| 구분 | 내용 |
|------|------|
| 해당되는 국가용 보안요구사항 | 6.1.1(업데이트 유효성 검증), 6.1.2(제품 유일 식별정보 확인), 6.1.3(실패시 기존버전 유지) |
| 보안기능 설명 | RAUC A/B 이중화로 업데이트한다. 번들은 서명되어 있으며 기기의 공개 인증서(keyring)로 검증한 뒤에만 설치한다. 설치 실패·부팅 실패 시 U-Boot bootcount가 이전 슬롯으로 자동 롤백한다. 제품 유일 식별정보(펌웨어 버전·빌드 해시·활성 슬롯)는 `fwinfo`로 확인한다. |
| 보안기능 동작 순서도 | 〈그림〉 |
| 보안기능의 동작 | (절차 1) 서명 번들 수신 → keyring으로 서명 검증, 실패 시 설치 거부. (절차 2) 비활성 슬롯에 기록 후 부트 우선순위 전환. (절차 3) 부팅 성공 확인 전까지 boot-attempts 카운트, 초과 시 이전 슬롯 롤백. |

*(구현 위치: `board/rpi5-secure/rootfs-overlay/etc/rauc/system.conf`, `.../usr/bin/fwinfo`)*
