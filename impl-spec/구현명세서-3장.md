# 보안기능 구현명세서 — 3장 기능 설명 (living)

이 문서는 **손으로 쓰는 서술(4단 골격)** + **자동 상태표**로 구성된다.
- 서술 부분은 언제든 직접 수정 가능(코드 작업 중에도).
- 맨 아래 `AUTO-STATUS` 블록은 `node sfr/sync.mjs --write` 가 코드의 `SFR:<id>` 태그를 스캔해 갱신한다(직접 수정 금지).
- 완성되면 rhwp-bridge 로 `.hwpx` 변환 → ① 구현명세서 3장에 삽입.

> 각 보안기능은 가이드 4단 골격으로: **해당 요구사항 / 설명 / 동작 순서도 / 동작 절차**.
>
> **구현 상태**: 암호(crypto_hal)·인증(auth)·감사(audit)·세션(session)·관리(mgmt/RBAC)·설정보호(config_store) 로직은 **실구현 + 호스트 테스트 통과**(`run-host-tests.sh`, `-Werror` 경고 0). 플랫폼 종속 기능(프로비저닝 파일·dm-verity 연동·관리 UI 렌더링·Buildroot 실빌드)은 골격 + TODO로 이식 시 완성.

## 3.2 보안기능 개요

제품(RPi5+IMX219, 순수 스트리밍)의 보안기능을 최소 단위로 식별. 접속평면·인증·영상보안·자체보호·감사·업데이트·암호로 구분하며, 각 기능은 아래 3.3에 개별 기술.

### 3.3.4 영상 프로토콜 인증 및 전송 보호 (SFR:1.1.1 / 1.2.1)

| 구분 | 내용 |

| 보안기능 설명 | IMX219 영상은 RTSP 서버(mediamtx)를 통해 스트리밍되며, 익명 접근을 차단하는 사용자 인증(internal auth)과 RTSPS(RTSP over TLS, `rtspEncryption: strict`)를 강제한다. 평문 RTSP·UDP 전송은 비활성화된다. RPi5는 HW 인코더가 없어 libcamera→gstreamer 소프트웨어 H.264로 인코딩한다. |

| 보안기능의 동작 | (절차 1) 클라이언트가 RTSPS(TLS)로 접속, 평문 RTSP는 거부. (절차 2) internal auth로 사용자(viewer/publisher) 인증, 실패 시 스트림 미제공. (절차 3) 인증된 세션에만 TLS 암호화된 영상 전송. (절차 4) 서버 인증서 부재 시 기동 보류(평문 폴백 금지). |
